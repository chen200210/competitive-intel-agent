## 🔴 严重 Bug (会导致数据丢失或功能缺失)

### BUG #1: 用户反馈计数器被每日清理重置

**文件**: `src/pipeline/runner.py:298`
**根因**: 两种不同生命周期的数据放在同一张表里

```python
# runner.py _run_phase0_scrape:
conn.execute("DELETE FROM market_news WHERE date = ?", (date,))
```

每次重跑 pipeline 都会 DELETE `market_news` 当天数据，然后 scrapers 重新 INSERT。这导致 `useful_count` 和 `useless_count` 列被重置为 0。

`increment_news_feedback()` (sqlite.py:1107) 已经意识到这个问题，加了注释 "market_news row might not exist"，但代码只做了静默吞异常，没有真正解决。

**影响链**:
1. 日报推送 → 用户点 👍/👎 → 计数器 +1
2. 同一天重跑 pipeline（比如 debug 或补数据） → `market_news` 被清空重建 → 计数器归零
3. `_build_feedback_summary()` JOIN `market_news` 查 headline → 如果该 URL 未被 scraper 重新抓到 → headline 为 NULL

**修复方向**: 
- 方案A: 清理时保留 `useful_count`/`useless_count`，重建后回填
- 方案B: 把计数器从 `market_news` 剥离，仅在 `user_feedback` 表聚合
- 方案C: 清理前先 dump 计数器到临时表，INSERT 后 UPDATE 回去

### BUG #2: 跨榜信号因列名错误而永远不显示

**文件**: `src/agents/briefer.py:282-293`
**根因**: SQL 查询了不存在的列，异常被静默吞掉

```python
cross_rows = db._connect().execute(
    "SELECT signals_json FROM cross_chart_signals WHERE date = ?",
    (date,)
).fetchall()
```

`cross_chart_signals` 表的实际列是：`charts_json`, `signal_pattern`, `signal_description`, `threat_level`。没有 `signals_json` 列。

SQL 会抛出 `sqlite3.OperationalError: no such column: signals_json`，被外层 `except Exception: pass` 吞掉，`cross_signals` 始终为空列表。

**对比 runner.py 的正确做法**:
```python
# runner.py 中通过 API 读取，不会出错：
from src.pipeline.cross_chart import get_signals_for_date
cross_signals = get_signals_for_date(date)
```

但 runner 读取的 cross_signals 只用于 track filter，并未传给 briefer。Briefer 独立从 DB 读，读错了。

**修复方向**:
- 改用 `get_signals_for_date(date)` 而非手写 SQL
- 或修正为 `SELECT * FROM cross_chart_signals WHERE date = ?` 然后用行数据构造信号

### BUG #3: market_news 永久 URL 去重阻止新闻重新出现

**文件**: `tools/scrapers/news_feeds.py:751-756`, `tools/scrapers/pocketgamer_biz.py:185-190`
**根因**: 两套去重系统的生命周期不一致

```python
# news_feeds._sync_to_db:
existing_urls = set()
rows = db._connect().execute(
    "SELECT DISTINCT url FROM market_news"  # ← 全部历史URL
).fetchall()
```

`market_news` 层面的去重是**永久的**：一旦 URL 入库，永远不再入库。但 `reported_items` 的去重有 TTL（news 30天，news_seen 7天）。

**场景**: 某游戏 3 个月前被报道"内测开启"，URL 入库。3 个月后它"正式上线"又被同一 URL 报道（很多游戏媒体会更新同一篇文章而非发新的），此时该新闻无法再入库，日报里永远不会出现。

**修复方向**:
- `_sync_to_db` 的去重应加时间窗口（如 30 天），与 `reported_items` 对齐
- 或直接移除 `_sync_to_db` 的去重，统一依赖 `reported_items` 层面的 TTL 去重

## 🟡 中等严重度 (潜在问题)

### BUG #4: `increment_news_feedback` 存在竞态条件

**文件**: `src/storage/sqlite.py:1087-1118`

去重检查 (`SELECT`) 和插入 (`INSERT`) 不在同一事务内：
```python
# 检查 (事务1)
existing = conn.execute("SELECT id FROM user_feedback WHERE ...")
if existing:
    return "duplicate"

# ... market_news 更新 ...

# 插入 (事务2)
self.insert_feedback(...)
```

两个并发的 👍 点击可能同时通过检查。虽然 `idx_user_feedback_dedup` 唯一索引会阻止重复插入，但 `insert_feedback` 用普通 INSERT（非 INSERT OR IGNORE），会抛 `IntegrityError` 被外层 `except` 捕获后返回 `"error"`。

**修复**: 用 `INSERT OR IGNORE` + `cursor.rowcount` 判断是否实际写入，在一个事务内完成。

### BUG #5: B站 来源判断使用裸字符串包含

**文件**: `src/agents/briefer.py:847-849`

```python
def _is_bili(item: dict[str, Any]) -> bool:
    src = (item.get("source", "") or "").lower()
    return "bilibili" in src
```

如果未来某个非 B站 来源名称包含 "bilibili" 字符串（比如 "bilibili-news" 或 URL 包含 bilibili.com），会误判。目前只有 `_bilibili_to_news` 设置 `source="bilibili"`，但这是脆弱设计。

**修复**: 精确匹配 `src == "bilibili"`，或在设置 source 时用常量。

### BUG #6: 反馈回复依赖 ephemeral 数据

**文件**: `src/feishu/bot.py:144-149`

```python
rows = db._connect().execute(
    "SELECT headline FROM market_news WHERE url = ? AND date = ?",
    (news_url, target_date),
).fetchall()
```

当 `market_news` 被清理后（同一天重跑），这条查询返回空 → 回复消息缺失新闻标题 → 用户体验差："感谢 xxx 的反馈，接下来将会推荐更多类似「」的新闻"。

**修复**: 如果 `market_news` 查不到，fallback 到 "相关内容" 或从 `user_feedback` 历史中推断。

## 🟢 代码质量问题

### #7: 静默异常吞没遍布全项目

以下位置使用 `except Exception: pass` 或等价模式，共 8+ 处：

| 文件 | 行号 | 影响 |
|------|------|------|
| briefer.py | 208 | DB upsert 失败静默 |
| briefer.py | 555 | TapTap URL 查找失败静默 |
| briefer.py | 1210 | track_filter 失败静默 |
| briefer.py | 1621 | fatigue 检查失败静默 |
| briefer.py | 1810 | AI summarizer 失败 fallback |
| sqlite.py | 1024 | mark_reported 失败静默 |
| sqlite.py | 1107 | counter update 失败静默 |
| runner.py | 272 | URL resolve 失败静默 |

**问题**: BUG #2 就是被这种模式藏住的。静默吞异常让 bug 难以发现，错误迹象完全消失。

**建议**: 
- 最低限度：`print(f"[WARN] {e}", file=sys.stderr)` 或 `logging.warning()`
- 更好：在关键路径（DB 写入、数据读取）不要吞异常

### #8: 两套去重系统互相不感知

| 层级 | 去重方式 | TTL |
|------|---------|-----|
| scraper `_sync_to_db` | 全表 URL 查重 | 永久 |
| briefer `_compact_news` | `reported_items` 表 | 30天(news) / 7天(news_seen) |

scraper 层永久阻止 URL 入库，但 briefer 层还在检查 `reported_items`。如果 briefer 想重新考虑一篇旧文章（因为 TTL 已过），它根本拿不到数据（因为 scraper 层永久去重了）。

**建议**: 统一去重策略 — scraper 只做当日去重，跨日去重全部交给 `reported_items`。

### #9: `brief_from_db` 与 `runner.py` 读取跨榜信号的方式不一致

- `runner.py` 通过 `get_signals_for_date()` API 读 → 正确
- `brief_from_db` 手写 SQL 读 → 列名错误 → 永远读不到

**建议**: `brief_from_db` 应该调用同一个 API，而不是重复实现读取逻辑。

## 📊 根因分析：为什么会产生这些 Bug

三个严重 bug 的共同模式：

1. **生命周期混叠** (BUG #1): 把"每天刷新"的数据（新闻内容）和"跨天累积"的数据（反馈计数）放在同一张表的同一行里
2. **静默吞异常** (BUG #2): 异常被吞掉后没有任何可见痕迹，功能悄悄失效
3. **重复实现** (BUG #2, #3, #8, #9): 同一个逻辑被实现了两遍（scraper 去重 vs briefer 去重；runner 读信号 vs briefer 读信号），两遍不一致
