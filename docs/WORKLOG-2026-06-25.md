# 工作日志 — 2026-06-25

## 一、RecGPT 论文研读与应用落地

### 论文理解

阅读阿里/淘宝 RecGPT 论文 (arxiv:2507.22879)，核心收获：

- **三塔召回架构**：User塔（协同过滤）+ Item塔（物品属性）+ Tag塔（LLM生成的商品标签），通过 `β × 协同分数 + (1-β) × 语义分数` 融合。LLM 提供语义增量，补充而非替代传统推荐。
- **三个 LLM 任务**：LLM_UI（用户兴趣挖掘）→ LLM_IT（商品标签预测）→ LLM_RE（推荐解释生成），每个遵循五段式 prompt 模板（Role → Input → Requirements → Matched Pool → Output）。
- **LLM-as-Judge 评估链**：人标注种子 → 微调 Judge → Judge 评估任务模型 → 人抽检 Judge 误判。
- **关键结论**：DeepSeek-R1 不微调也能用（标签通过率 80%，微调后最优仅 88.8%），中小团队 20% 成本完成 80% 效果。

提炼出五个跨场景通用的工业设计模式：

1. **Matched Interest Pool**（输出空间约束）— 预设合法标签池，约束 LLM 不能发明分类
2. **CoT 显式拆分**（任务分解）— 主动拆分任务为多步，而非让 LLM "自己一步步想"
3. **β-Fusion**（信号融合）— 代码规则信号与 LLM 语义信号加权融合，互相补充
4. **LLM as a Judge**（评估链）— 用 LLM 评估 LLM 的输出质量，人只监督 Judge
5. **离线预计算**（LLM 离线干活）— LLM 结果持久化，在线零成本读取

### 五个模式全部落地至日报系统（P0-P4）

| 优先级 | 模式 | 改动 | 状态 |
|--------|------|------|------|
| P0 | LLM as a Judge → Calibrator | `calibrator.py` + `calibrator.yaml` + DB table + scorer 集成 | ✅ |
| P1 | β-Fusion | `scorer.py`：`_compute_signal_score()` → `0.3×signal + 0.7×AI` 融合 | ✅ |
| P2 | Matched Interest Pool | 6+6 Matched Verdict Pool + 标签持久化 + Calibrator 交叉分析 | ✅ |
| P3 | 离线预计算 | Briefer AI 排版调用移除，market_md 全部代码直拼 | ✅ |
| P4 | CoT 显式拆分 | `summarizer.yaml` 5 步推理链（信息提取→摘要→打分→标签→输出） | ✅ |

**LLM 调用点从 3 个缩减为 1 个**（Summarizer），其余全部走代码或缓存。

### 关键技术决策

- **β=0.3 而非 RecGPT 的 0.5**：我们的传统方法（代码正则）可靠性远低于淘宝协同过滤，AI 应占主导。但 0.3 能防止离谱打分（空洞通稿打 75 → 融合后 57）。
- **Judge 不需要微调**：Calibrator 只需分析 14 个话题 × 1 维度（好评/差评），prompt + 反馈统计足够。RecGPT 需要微调是因为评估 3000+ 标签 × 4 维度。
- **标签池 6+6 而非 3000+**：日报编辑的判断维度不需要那么多标签。更多标签反而降低 LLM 选择准确率（选择瘫痪）。
