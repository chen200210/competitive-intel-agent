## Part 1: 做得好的 (Keep Doing)

### 1. "代码预制，AI 只做增量" — 架构判断力 ★★★★★

AI 只输出 `market_md` 一段字符串，新游列表和排名表格全部由 Python 代码生成。不会出现表格格式错乱、链接幻觉、数据对不齐。这是 AI 工程中最核心的判断力——知道哪里该用 AI、哪里不该用。

**Keep**: 保持这个边界。未来加新板块时，**永远先问自己"这段内容能不能用代码生成"**，而不是直接把数据扔给 LLM。

### 2. 四阶段新闻流水线 — 成本/质量平衡 ★★★★☆

`规则粗筛(15)→正文深挖→AI 打分摘要→AI 排版`。30-50 条新闻 → 规则过滤到 15 条（省 60%+ AI 调用）→ 只有 top 7 做排版（再省 50%）。Sumarizer 批量处理 15 条而非逐条调用。

**Keep**: 加新新闻源时，**必须在 Phase A（规则粗筛）对新源做关键词覆盖**，确保劣质内容不会流入 AI 管线。

### 3. 三层去重体系 — 工程完备性 ★★★★☆

`reported_items`(30 天 TTL) + `insert_market_news_deduped`(同日窗口) + `news_seen`(7 天 TTL)。三个 TTL 窗口各司其职，低分候选短期屏蔽但不永久封杀，高分推送长期去重。

**Keep**: 加新内容类型（如新 scraper）时，**必须同步定义其 item_type 和 TTL 策略**，不允许绕过 `reported_items` 表直接入库。

### 4. CLAUDE.md 迭代记录 — 自主进化能力 ★★★★☆

4 轮架构升级记录从跨榜信号移除→GBK 兼容→静默异常消灭→去重策略统一。每个 fix 都是生产环境才会暴露的真问题。**"发现问题→修根因→留记录"的闭环**胜过任何花哨的架构图。

**Keep**: 每次较大改动后更新 CLAUDE.md。未来如果团队扩到 2-3 人，这些记录会是最宝贵的 onboarding 材料。

### 5. Scraper 基类的 Template Method 模式 ★★★☆☆

`ChartScraper.run() → scrape()[子类实现] → _clean()[基类] → _write_csv()[基类]`。新增数据源只需实现 `scrape()`，字段映射和 CSV 落盘统一处理。

**Keep**: 新增 scraper 时严格遵守此模式，不要另起炉灶。

### 6. JSON 解析的 5 层容错 ★★★☆☆

`json.loads → JSON 修复 → markdown fence 剥离 → 正则提取 → 兜底错误字典`。LLM 的 JSON 输出确实不可靠，这是必要的防御。

### 7. 纯规则赛道分类 + 故事检测 — 0 Token 成本 ★★★★☆

`track_filter.py` 和 `story_picker.py` 完全不消耗 AI token。关键词匹配 + 优先级链解决了分类问题，5 类故事检测也全是纯 Python。

### P2 — 错误处理策略不统一

`briefer.py` 中对 DB 操作有的 `try/except + print` 继续跑，有的静默 return 空集合。DB 写失败只打 WARN 然后继续，但卡片没有持久化意味着后续推送拿不到数据。

**行动计划**:

1. 定义三类失败级别：
   - **FATAL**: 阻断流水线（DB 无法连接、所有 scraper 失败、Briefer 无输出）
   - **WARN**: 不影响核心输出（图片上传失败、单条新闻正文抓取超时、反馈计数写失败）
   - **INFO**: 不影响任何输出（缓存未命中、候选被去重过滤）

2. `runner.py` 的 `_step()` 已经区分了 ok/skipped/error，但 error 后继续跑。改为：
   - FATAL 级别的 error → 立即 `sys.exit(1)`
   - WARN 级别的 error → 记录到 `pipeline_runs` 表，飞书推送附带警告摘要

3. 所有 `except Exception: return set()` 改为至少打一行 WARN，除非你明确知道该场景可忽略。

**验收标准**: 任何 DB 写失败不会被静默吞掉，`runner.py` 的 exit code 反映真实成功/失败状态。

### P5 — 缺少监控和告警

系统依赖人工看 stderr 输出感知问题。定时调度跑起来后没人看终端。

**行动计划**:

1. **`pipeline_runs` 表** — 每次 runner 写一行，记录各 Phase 的 success/fail + elapsed_ms + error_msg
2. **`runner.py` exit code** — 任何 FATAL 错误 → `sys.exit(1)`，全成功 → `sys.exit(0)`
3. **飞书推送附带健康摘要** — 如果有 WARN，在日报卡片最后加一个折叠的"系统状态" note，"今日全部正常"或"⚠️ B站抓取失败，不影响其他板块"

```sql
CREATE TABLE IF NOT EXISTS pipeline_runs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    date TEXT NOT NULL,
    started_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    phases_json TEXT,  -- [{"phase": "0A", "status": "ok", "elapsed_ms": 45000}, ...]
    exit_code INTEGER,
    error_summary TEXT,
    total_ms INTEGER
);
```

**验收标准**: 跑完流水线后 `pipeline_runs` 表有一行记录，飞书卡片末尾有健康摘要。
