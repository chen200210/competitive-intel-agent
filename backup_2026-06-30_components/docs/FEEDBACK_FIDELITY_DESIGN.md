## 一、概念：反馈 ≠ 真相

### 1.1 一个具体场景

日报推送了一条新闻：

> "《明日方舟》新肉鸽模式上线，首周玩家破百万"

用户点了 👍。

这个 👍 **到底意味着什么**？至少有 8 种可能：

| # | 用户真实心理 | 这条 👍 应该驱动 Calibrator 调参吗？ |
|---|------------|-----------------------------------|
| 1 | 用户觉得这条新闻有价值，希望多推送类似内容 | ✅ 应该 — 这是真正的偏好信号 |
| 2 | 用户喜欢明日方舟这个游戏，看到它就开心 | ⚠️ 部分应该 — 偏好"塔防+肉鸽"类别，但并非偏好"所有明日方舟新闻" |
| 3 | 用户是鹰角网络员工，支持自家产品 | ❌ 不应该 — 利益冲突，不代表目标受众偏好 |
| 4 | 用户前天刚和同事聊到肉鸽模式，碰巧看到 | ❌ 不应该 — 偶发兴趣，不具持续性 |
| 5 | 用户没细看，每天早上习惯性给日报点一圈 👍 | ❌ 不应该 — 习惯性行为，零信息量 |
| 6 | 用户拇指滑屏时不小心碰到了按钮 | ❌ 不应该 — 纯粹的噪声 |
| 7 | 用户觉得 AI 摘要写得不错，在赞 AI 而不是新闻 | ❌ 不应该 — 反馈目标错位 |
| 8 | 用户对上一条新闻很生气，但点错了按钮 | ❌ 不应该 — 噪声，且方向反了 |

**所有这 8 种情况，在数据库里都是完全相同的记录：**

```sql
INSERT INTO user_feedback (news_url, feedback_type) VALUES ('...', 'thumbs_up');
```

看不出任何区别。

### 1.2 根本困境

```
你关心的：用户真正的偏好（原因 1）
你能观测的：用户的点击行为（原因 1-8 的混合体）

两者之间的差距 = 失真
```

保真度（fidelity）衡量的是：

> **这条反馈，有多大概率是"用户真正的偏好"（原因 1），而非噪声（原因 3-8）？**

### 1.3 为什么这个差距会自我放大

假设你的日报有 3 个用户，反馈数据如下：

| 日期 | 新闻 | AI 打分 | 赛道相关 | 👍 | 👎 | 实际含义 |
|------|------|---------|---------|----|-----|---------|
| D1 | 明日方舟新肉鸽上线 | 72 | ✅ 塔防+肉鸽 | 5 | 0 | 5 个不同用户真心觉得好 |
| D2 | 某二次元手游日服营收破亿 | 22 | ❌ 二次元（排除赛道） | 4 | 0 | **同一用户**连续 4 天习惯性点赞 |
| D3 | 塔防游戏设计趋势分析 | 68 | ✅ 塔防 | 0 | 0 | 用户没注意（但内容质量高） |
| D4 | 独立游戏获苹果推荐 | 55 | ⚠️ 无关 | 1 | 0 | 单个用户随手点 |
| D5 | Steam移植技术分享 | 65 | ✅ Steam移植 | 0 | 1 | 单个用户手滑误触 |

**如果 Calibrator 不加区分地学习所有反馈：**

```
D2 的 4 个 👍（噪声） > D1 的 5 个 👍（真实信号）
→ topic_boosts.二次元 +10
→ 明天日报推送更多二次元新闻
→ 其他 4 个用户觉得垃圾变多了
→ 他们停止点 👍（核心用户流失）
→ Calibrator 收到的信号越来越偏
→ 恶性循环
```

这就是 **self-consuming model collapse**（自我吞噬的模型崩塌）——RSIR 论文的核心命名对象。

## 二、保真度是什么

### 2.1 一句话定义

> **保真度 = 从可观测的旁证推断出"这条反馈代表用户真实偏好"的概率**

这是一个**概率推断**问题。你永远不知道真相（用户心里到底怎么想的），但你可以根据旁证（circumstantial evidence）估算概率。

### 2.2 类比：法庭判案

| 证据类型 | 推荐系统的对应 | 高保真意味着 | 低保真意味着 |
|---------|--------------|-------------|-------------|
| 目击者证词 | 用户点击了 👍 | — | — |
| 物证 | 用户也点开了原文链接 | 用户真的看了内容 | 用户可能只看了标题 |
| 多名独立证人 | 3 个不同用户都 👍 | 多人独立确认 | 重复可能是刷票 |
| 证人与案件关系 | 用户是目标受众（赛道游戏） | 专业判断 | 利益冲突 |
| 证词间一致性 | AI 打了高分 + 用户 👍 | 两个独立系统得出相同结论 | — |
| 证词间矛盾 | AI 打了低分 + 用户 👍 | 可能发现 AI 盲区 | 大概率是用户手滑 |
| 证人可信度 | 该用户历史反馈是否一致 | 一贯有判断力的用户 | 随机点击的用户 |
| 证据浓度 | 3👍:0👎 vs 1👍:1👎 | 共识明确 | 分歧大 = 信号弱 |

保真度分数 = 综合所有这些旁证，给每条反馈一个 0-1 的信度估计。

### 2.3 三条反馈的对比

#### 高保真反馈 (fidelity ≈ 0.85)

```
新闻: "《明日方舟》新肉鸽模式上线，玩家好评如潮"
AI 打分: 72 (track_direct + high_info_density)
赛道相关: ✅（塔防+肉鸽双命中）
反馈: 👍👍👍 (3 个不同用户)
用户行为: 2 人点开了原文链接
标签: pos_label="track_direct"

综合判断: AI 和 3 个用户独立达成一致，赛道相关，有原文访问佐证
→ 结论: "用户偏好塔防+肉鸽类深度报道" — 高度可信
→ Calibrator 应该学习
```

#### 低保真反馈 (fidelity ≈ 0.15)

```
新闻: "某二次元手游日服营收破亿"
AI 打分: 22 (off_track + generic_pr)
赛道相关: ❌（二次元 → 明确排除赛道）
反馈: 1👍, 0👎
标签: neg_label="off_track"

综合判断: AI 判断 off-track，与赛道规则一致。单用户孤证，无原文访问
→ 结论: "用户可能手滑/习惯性点赞/误读标题" — 概率远高于"用户想要二次元新闻"
→ Calibrator 不应该学习
```

#### 看似"负面"实则高保真的反馈 (fidelity ≈ 0.70)

```
新闻: "凉屋新作《XXX》Steam移植版上线，融合塔防+肉鸽"
AI 打分: 35（AI 误判为 generic_pr，实际上内容很好）
赛道相关: ✅（塔防+肉鸽）
反馈: 👎👎👎 (3 个不同用户 👎)
用户行为: 3 人都看了原文

综合判断: 多人独立 👎 + 赛道相关 + AI 低分，但用户看了原文说明他们认真对待
这不是"用户不喜欢赛道新闻" — 更可能是"AI 摘要没抓住重点，或标题误导"
→ 结论: AI 可能低估了这类新闻 → AI-反馈分歧信号，极其珍贵
→ Calibrator 应该学习，但学习的是"收紧 track_direct 标签标准"，而非"降低塔防/肉鸽权重"
```

**第三条是关键**：保真度高不等于方向正。AI-反馈分歧是最有价值的校准信号——它告诉你模型哪里判断错了。

## 三、为什么 Calibrator 需要它

### 3.1 现状：Calibrator 收到的是原始反馈

```
user_feedback (所有 👍/👎)
    │
    ▼ _aggregate_feedback() — SQL 聚合，不加过滤
    │
    ▼ _format_feedback_table() — 格式化
    │
    ▼ Calibrator LLM — 分析所有反馈
    │
    ▼ topic_boosts + dim_weights
```

**问题**：Calibrator 看到的每条反馈都是平等的。D1 的 5 个高价值 👍 和 D2 的 4 个噪声 👍，在表格里长得一样。

### 3.2 Calibrator prompt 里已有的防御不够

Prompt 里已经写了：

> 是否有 ≥3 条独立反馈支撑？（单条反馈是噪声）

但这是**靠 LLM 自觉**——LLM 能看到的只有 `| GameLook | 3 | 0 | 👍偏好 | | | | 某二次元手游... |`，它无法知道：

- 这 3 个 👍 可能来自同一个人
- AI 给这条新闻打了 22 分（明确 off-track）
- 这条新闻属于明确排除的二次元赛道

**这些上下文字段不在 prompt 里。**

### 3.3 插入点

保真度检查应该插入在 `_aggregate_feedback()` 和 `_format_feedback_table()` 之间：

```
_aggregate_feedback()  →  拿到所有反馈（包括噪声，如 D2）
    ↓
【新增】_compute_fidelity(row)  →  给每条反馈算保真度
【新增】filter(fidelity >= threshold)  →  扔掉低保真度的（D2 被丢弃）
    ↓
_format_feedback_table()  →  只把高保真度的喂给 LLM
```

### 3.4 改动范围

**仅 `src/agents/calibrator.py` 一个文件**。不需要新表、新字段、新 prompt、新依赖。原因：

- `_aggregate_feedback()` 的 SQL 已经 join 了 `user_feedback` + `market_news`，返回了所有需要的字段
- `_compute_fidelity()` 是一个纯 Python 函数，零 AI 成本，结构完全对标 `scorer.py` 里已有的 `_compute_signal_score()`
- 阈值过滤是一条 `if fidelity < threshold: continue`

## 四、RSIR 论文的概念嫁接

### 4.1 三个核心概念的翻译

RSIR 论文的数学语言 → 日报系统的工程语言：

| RSIR 概念 | 论文定义（简） | 日报系统对应 | 具体含义 |
|-----------|-------------|-------------|---------|
| **偏好流形** (Preference Manifold) | 所有"合理"的用户偏好构成低维曲面，嵌在高维行为空间中 | 符合赛道规则 + AI 判断 + 多人共识的反馈集合 | 例：塔防肉鸽新闻被多个独立用户 👍 — "在流形上" |
| **切空间隐式正则化** (Tangent Space Implicit Regularization) | 保真度过滤使参数只沿流形切方向更新，不被噪声拉偏 | 高保真反馈驱动 Calibrator 调参，低质量反馈不参与 | 例：`topic_boosts.独立游戏 +5`（沿流形），不因噪声调整 `topic_boosts.二次元`（偏离流形） |
| **自我吞噬崩塌** (Self-Consuming Model Collapse) | 合成样本中的低保真度漏网之鱼逐代累积，输出退化 | 噪声反馈 → Calibrator 误调参 → 日报质量下降 → 核心用户流失 → 更偏的反馈 → 循环 | 例：二次元加权 → 推更多二次元 → 用户流失 → 只剩一个用户的反馈 → 完全偏斜 |

### 4.2 切空间正则化的直觉

这是论文里最难懂的概念，用你的系统来翻译：

```
流形 = 所有"讲得通"的 (新闻类型, 用户偏好) 组合构成的一个曲面

切空间 = 在这个曲面上，你可以"合理移动"的方向：
  - 更多独立游戏深度报道 → "在流形上移动" ✅
  - 更多塔防肉鸽新闻 → "在流形上移动" ✅
  - 更多海外市场分析 → "在流形上移动" ✅

垂直于流形的方向 = "不合理的移动"：
  - 更多二次元手游营收新闻（明确排除的赛道）→ "跳出流形" ❌
  - 更多低质通用 PR 稿（被所有用户 👎 的内容）→ "跳出流形" ❌
  - 更多 AI 打 22 分但被莫名其妙 👍 的新闻 → "跳出流形" ❌

保真度过滤 = 丢弃所有"垂直于流形方向"的反馈信号
→ Calibrator 的参数更新只沿着"合理方向"走
→ 等价于切空间隐式正则化
```

**一句话：保真度检查让 Calibrator 的 `topic_boosts` 只往"讲得通的方向"调整，不会被噪声拉着往奇怪的方向走。**

### 4.3 和 Scorer 强制分布的对称性

你已经做过的"强制分布"（≥25% <40 分 + ≤30% >60 分）和保真度检查是**同一个思想在两个对称位置的应用**：

```
                    Scorer 侧                          Calibrator 侧
                    ─────────                         ─────────────
问题         AI 打分塌缩到中间段                 噪声反馈污染参数更新
（模型输出漂移）                                  （反馈输入质量下降）

解法         强制分布检查 + fallback              保真度检查 + 阈值过滤
（外部约束）  （代码层纠正 AI 输出）               （代码层过滤 LLM 输入）

位置          scorer.py: _check_distribution()     calibrator.py: _compute_fidelity()
             scorer.py: _apply_score_fallback()    calibrator.py: filter(fidelity >= 0.3)

目标         防止 AI 输出漂移到低质量均衡         防止 Calibrator 输入被噪声污染
```

这两个机制是互补的：
- 强制分布 → **输出**侧的质量门禁
- 保真度过滤 → **输入**侧的质量门禁

两者一起形成完整的反馈闭环质量保障。

## 五、信号设计

### 5.1 概述

`_compute_fidelity(row: dict) -> float` 返回 0.0–1.0 的信度分数。

四个信号，按重要程度排列：

| # | 信号名 | 权重 | 含义 | 数据来源 |
|---|--------|------|------|---------|
| 1 | AI-反馈一致性 | **40%** | AI 判断和用户反馈是否指向同一方向 | `pos_label` / `neg_label` + `sentiment` |
| 2 | 反馈浓度 | **30%** | 反馈是共识还是分歧 | `up` / `down` |
| 3 | 赛道锚定 | **20%** | 反馈对象是否在关注域内 | `track_relevant` |
| 4 | 标签覆盖 | **10%** | 是否可做 label×feedback 交叉分析 | `pos_label` / `neg_label` 非空 |

**设计原则**：
- 信号 1（AI-反馈一致性）权重最高 — 它是最强的旁证。AI 和用户是两个独立系统，它们的一致/分歧是判断反馈真伪的核心线索。
- 信号 2（反馈浓度）次之 — 多个独立用户的一致判断比单用户孤证可靠。
- 信号 3（赛道锚定）是领域知识 — 你的系统有明确的关注域（塔防/肉鸽/割草），在域内的反馈比在域外的反馈更可信。
- 信号 4（标签覆盖）权重最低 — 它只是锦上添花，让下游的 label×feedback 交叉分析更准确。

### 5.2 信号 1：AI-反馈一致性（40%）

**直觉**：AI 打分和用户反馈是两个独立系统对同一条新闻的判断。它们的一致性/分歧是判断反馈真伪的最强信号。

**计算**：

```
反馈极性 = +1 (👍偏好) / -1 (👎排斥) / 0 (中性)

AI 质量判断 = 
  +1 如果 pos_label ∈ {track_direct, high_info_density, exclusive_scoop, overseas_insight}
  -1 如果 neg_label ∈ {off_track, no_gameplay, low_info, stale}
   0 其他

一致性 = feedback_polarity × ai_quality_judgment

一致性 = +1  → "AI 看好，用户也 👍" — 高保真，两个独立系统达成共识
一致性 = -1  → "AI 看好，用户 👎"（或反过来）— 中等保真，分歧信号本身有价值
一致性 =  0  → "AI 中性，用户反馈" — 中性
```

**为什么"分歧"不是低保真**：AI-反馈分歧（AI 高分 + 用户 👎，或 AI 低分 + 用户 👍）是 Calibrator 最需要仔细分析的情况。它不是"噪声"——它可能是：
1. AI 盲区（AI 低估了某类内容的价值）
2. 用户误解（用户误读了标题或摘要）
3. 真正的偏好分歧（赛道用户和非赛道用户的判断不同）

保真度分数对"分歧"给中等分数（不高也不低），交给 Calibrator LLM 做深度分析。

**fidelity 贡献**：

| 场景 | AI-反馈关系 | 含义 | fidelity 贡献 |
|------|-----------|------|:---:|
| AI track_direct + 用户 👍👍👍 | 强一致 | 强烈确认赛道偏好 | **+0.40** |
| AI high_info_density + 用户 👍 | 一致 | 用户认可深度内容 | **+0.30** |
| AI off_track + 用户 👍👍👍 | **强矛盾** | 可能 AI 盲区或用户误读 | **+0.15** （中等，需 LLM 分析） |
| AI low_info + 用户 👍 | 矛盾 | 偏差信号，但 weaker | **+0.10** |
| AI track_direct + 用户 👎👎👎 | **强矛盾** | 可能 AI 标签过松 | **+0.15** （中等） |
| 无标签 + 用户 👍 | 无参照 | AI 未给出明确判断 | **+0.05** |

### 5.3 信号 2：反馈浓度（30%）

**直觉**：多个独立用户的共识比单用户孤证可靠。"3👍:0👎" 和 "1👍:1👎" 的信息量完全不同。

**计算**：

```python
total = up + down
if total == 0:
    return 0.0  # 不应该出现，但防御

ratio = abs(up - down) / total  # 0.0（均分）→ 1.0（全赞或全踩）

# 基础分：浓度 × 参与度
# total >= 5: 高参与度，浓度可信
# total >= 3: 中等参与度
# total == 1: 最低参与度（单用户孤证）
```

| 场景 | up | down | total | ratio | 含义 | fidelity 贡献 |
|------|----|------|-------|-------|------|:---:|
| 强共识，高参与 | 5 | 0 | 5 | 1.0 | 5 人一致 👍，非常可靠 | **+0.30** |
| 强共识，中参与 | 3 | 0 | 3 | 1.0 | 3 人一致 👍，可靠 | **+0.25** |
| 强共识，低参与 | 1 | 0 | 1 | 1.0 | 单用户孤证，不可靠 | **+0.05** |
| 弱共识，高参与 | 3 | 2 | 5 | 0.2 | 5 人意见严重分歧，信号弱 | **+0.05** |
| 弱共识，中参与 | 2 | 1 | 3 | 0.33 | 3 人意见分歧 | **+0.08** |

**关键**：1 个 👍 即使 ratio=1.0（100% 正向），也不能给高分 — 因为 total=1 意味着单用户孤证。这就是为什么不能直接用 `up / (up + down)` 做保真度。

### 5.4 信号 3：赛道锚定（20%）

**直觉**：你的系统有明确的关注域（塔防/肉鸽/割草）。在关注域内的反馈，用户是目标受众，判断更可信。在关注域外的反馈，更可能是手滑或误解。

**计算**：

```
track_relevant = True  → +0.20
track_relevant = False → +0.00
```

这是领域知识注入点。不能给负面分 — 赛道外的反馈本身不是"假的"，只是少了一个可信度锚点。

### 5.5 信号 4：标签覆盖（10%）

**直觉**：AI 给出了明确的 pos_label/neg_label 标签，意味着这条新闻有质量锚定，后续 Calibrator 可以做 label×feedback 交叉分析。

**计算**：

```
has_label = (pos_label 非空) 或 (neg_label 非空)
has_label → +0.10
无标签    → +0.00
```

### 5.6 综合公式

```python
def _compute_fidelity(row: dict) -> float:
    score = 0.0

    # 信号 1: AI-反馈一致性 (40%)
    score += _signal_ai_feedback_consistency(row) * 0.40

    # 信号 2: 反馈浓度 (30%)
    score += _signal_feedback_concentration(row) * 0.30

    # 信号 3: 赛道锚定 (20%)
    score += _signal_track_anchor(row) * 0.20

    # 信号 4: 标签覆盖 (10%)
    score += _signal_label_coverage(row) * 0.10

    return round(score, 3)
```

### 5.7 阈值设计

| 阈值 | 含义 | Calibrator 行为 |
|------|------|----------------|
| fidelity ≥ 0.60 | 高保真 — 强烈信号 | 喂给 LLM，重点分析 |
| 0.30 ≤ fidelity < 0.60 | 中等保真 — 有信号但不确定 | 喂给 LLM，标记 `[fidelity: medium]` 提示谨慎 |
| fidelity < 0.30 | 低保真 — 大概率噪声 | **不喂给 LLM**，丢弃 |

**阈值 0.30 的选择**：
- 单用户孤证 + 非赛道 + 无标签 → fidelity ≈ 0.05（远离阈值，被丢弃）
- 单用户孤证 + 赛道相关 + 有标签 → fidelity ≈ 0.35（刚过阈值，标记 medium）
- 3 人一致 👍 + 赛道相关 + AI 一致 → fidelity ≈ 0.90（远超阈值）

### 5.8 标记而非强制丢弃（中等保真度）

对于 fidelity ∈ [0.30, 0.60) 的反馈，不直接丢弃，而是在格式化时添加提示：

```
| GameLook | 3 | 0 | 👍偏好 | 🏷️ | track_direct | | 某独立游戏... | [fidelity: medium] |
```

让 Calibrator LLM 知道这条反馈的置信度有限，但仍保留它做分析。这给了 LLM 一个"谨慎对待"的提示，而非完全剥夺信息。

**只有 fidelity < 0.30 的才直接丢弃** — 这些几乎可以确定是噪声（单用户孤证 + 非赛道 + 无标签，或 1👍:1👎 分歧 + 非赛道）。

## 六、实现现状

### 6.1 已实现（2026-06-26）

**仅 `src/agents/calibrator.py` 一个文件**，新增 ~140 行，零新依赖。

```
src/agents/calibrator.py
  ├── 模块级常量
  │   ├── _FIDELITY_DROP = 0.30        # 低于此 → 丢弃
  │   ├── _FIDELITY_MEDIUM = 0.60      # 低于此 → 保留但标记
  │   ├── _SENTIMENT_UP / _DOWN / _NEUTRAL  # 情感字符串单一真相源
  │   ├── _AI_POSITIVE_LABELS          # 强正面标签（scorer.py 的有意子集）
  │   └── _AI_NEGATIVE_LABELS          # 强负面标签
  ├── 4 个信号函数（纯函数，零 token）
  │   ├── _signal_ai_feedback_consistency(row)  # 40% — AI 与用户的一致性
  │   ├── _signal_feedback_concentration(row)   # 30% — 参与深度 × 共识度
  │   ├── _signal_track_anchor(row)             # 20% — 赛道锚定
  │   └── _signal_label_coverage(row)           # 10% — 标签覆盖
  ├── _compute_fidelity(row) → float           # 加权融合
  ├── _aggregate_feedback() — 集成过滤        # fidelity < 0.30 → 丢弃
  ├── _format_feedback_table() — 集成标注     # 0.30~0.60 → `[信度:x.xx]`
  └── run_calibrator() — 集成日志             # verbose 模式输出过滤统计
```

### 6.2 与设计文档的差异

| 设计项 | 原设计 | 实际实现 |
|--------|--------|---------|
| 浓度公式 | `ratio * depth`（乘法） | `depth * (0.5 + 0.5 * ratio)`（参与度主导，共识度 ±50% 调制） |
| 情感字符串 | 硬编码在 `_aggregate_feedback` 和 `_signal_ai_feedback_consistency` 两处 | 模块级常量 `_SENTIMENT_UP/DOWN/NEUTRAL`，两处共享 |
| 标签集 | 无文档，独立维护 | 有注释说明与 `scorer.py._VALID_*_LABELS` 的关系，标注了有意排除的标签 |
| 阈值跳过消息 | "仅 N 条反馈，不足阈值" | "仅 N 条高信度反馈（经保真度过滤），不足阈值" |
| 代码组织 | 伪代码框架 | 完整实现 + 语法验证 + 参考点验证 |

### 6.3 Code Review 发现与修复（2026-06-26）

| # | 发现 | 严重程度 | 修复 |
|---|------|:---:|------|
| 1 | `ratio * depth` 导致 3-2 分歧（5 人，更有价值）得分 0.20 低于 2-0 一致（2 人）得分 0.40 | 高 | 改为 `depth * (0.5 + 0.5 * ratio)`，3-2 现在 0.60 > 2-0 的 0.40 |
| 2 | 情感字符串 `"👍偏好"` 在两处独立硬编码，一处改动静默破坏保真度 | 高 | 提取为 `_SENTIMENT_UP/DOWN/NEUTRAL` 常量 |
| 3 | `_AI_POSITIVE_LABELS` 是 `scorer.py._VALID_POS_LABELS` 的独立维护子集，4 个标签缺失且无文档 | 中 | 添加注释说明有意排除关系 |
| 4 | 跳过消息未提及保真度过滤，操作者看到 "仅 25 条反馈" 却不知原始有 40 条 | 中 | 改为 "仅 25 条高信度反馈（经保真度过滤）" |

## 七、当前缺口

保真度检查的代码骨架是完整的，但它本质上是一个**"用未经验证的规则去过滤另一个规则（Calibrator）的输入"的元系统**。以下缺口使其尚未真正闭环：

### 7.1 未在真实数据上运行

`_FIDELITY_DROP = 0.30` 和 `_FIDELITY_MEDIUM = 0.60`，以及四个权重 `0.40/0.30/0.20/0.10`，全部来自推理而非数据。

**实际情况未知**：
- 真实反馈的保真度分布长什么样？80% 被过滤还是 5%？
- 被丢弃的反馈里有没有被误杀的有价值信号？
- 阈值的"手感"对不对？

### 7.2 Calibrator prompt 不理解 `[信度]` 标记

`_format_feedback_table()` 在中等保真度条目的标题列追加了 `` `[信度:0.41]` ``，但 `prompts/calibrator.yaml` 没有告诉 LLM 这个标记的含义。LLM 大概率会无视它。

### 7.3 没有验证闭环

保真度过滤是否真的让 Calibrator 输出变好？目前没有任何度量。无法回答：
- 过滤前 vs 过滤后，`topic_boosts` 有什么不同？
- 哪个版本的参数让后续日报的用户反馈更好？
- 有没有被误杀的高价值反馈？

### 7.4 过滤统计仅在 stderr，无法编程获取

`_aggregate_feedback()` 将丢弃数打印到 stderr 但不在返回值中携带。如果保真度过滤太激进导致反馈全被丢弃，调用者只能从日志中手动发现。

### 7.5 这不是真相检测器

保真度检查不做"这条反馈是对是错"的判断。它只做概率推断。它**不能**：
- 防止恶意刷票（需要 rate limiting / 用户身份验证）
- 纠正 AI 的系统性偏见（如果 AI 系统性地给某类新闻打低分，保真度会继承偏见的某些方面）
- 替代人工审核（极端情况仍需人看）

## 八、下一步

### Step 1 — 真实数据验证（最小可行）

等 `user_feedback` 积累到 50+ 条后：

```bash
python -m src.pipeline.runner --calibrate --calibrate-days 30 -v
```

观察 stderr 输出：
- 保真度过滤丢了多少？保留了多少？
- 抽查 3-5 条被丢弃的反馈 — 是真的噪声还是被误杀？
- 抽查 3-5 条保留的反馈 — 保真度分值和直觉一致吗？

**决策点**：
- 丢弃 >60% → `_FIDELITY_DROP` 太高，下调到 0.20
- 丢弃 <5% → 阈值太低，没起过滤作用，上调到 0.40

### Step 2 — 让 Calibrator 理解信度标记

在 `prompts/calibrator.yaml` 的 Input 段添加：

```yaml
# 在 user_template 或 system prompt 中追加：
反馈表格中带有 `[信度:x.xx]` 标记的条目表示该反馈的保真度中等（0.30-0.60），
即证据链不够完整（如单用户孤证、非赛道领域、AI未给出明确标签等）。
对这些条目请降低分析权重，优先采信无标记的高保真度条目。
无标记的条目保真度 > 0.60，可放心参考。
```

不改这一步，`[信度]` 标记等于白标注。

### Step 3 — 手工 A/B 验证

Calibrator 跑完后，用 `--force` 跑一次不使用保真度过滤的对照版本，对比两组的 `topic_boosts` 和 `findings`：

```bash
# A 组：保真度过滤（默认）
python -m src.agents.calibrator --days 30 -v

# B 组：跳过保真度（需要临时修改 _FIDELITY_DROP = 0 或加 --no-fidelity 参数）
```

人工对比：
- 哪组的 topic_boosts 更合理？
- 过滤掉的反馈如果纳入，会不会导致 Calibrator 做出奇怪判断？
- 有没有 Calibrator 本应发现但被过滤阻断的模式？

### Step 4 — 长期迭代

以下在 Step 1-3 验证通过后考虑：

1. **阈值数据驱动** — 根据真实保真度分布，用分位数而非固定值设阈值
2. **用户级保真度** — 跟踪每个用户的反馈历史，给"一贯有判断力"的用户更高权重
